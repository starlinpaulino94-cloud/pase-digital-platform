# CAPÍTULO 4 — UX (CRÍTICA DURA Y NUEVO SISTEMA DE EXPERIENCIA)

*Auditoría Maestra de MembeGo · Volumen 4*

---

## 4.1 Diagnóstico general

La UX de MembeGo es la de un producto construido *módulo por módulo*: cada sección individual es razonable, pero el conjunto carece de **jerarquía de intención**. El usuario (cliente o empresa) entra y encuentra un menú de todo, no un camino hacia algo. Los cuatro pecados dominantes:

1. **Paridad de jerarquía:** todo pesa lo mismo. En el panel de empresa, "Configurar FAQ" y "Ver cuánto dinero generaste" tienen el mismo tamaño de link en el mismo sidebar.
2. **Vocabulario interno filtrado al usuario:** "Campañas", "Estrategias", "Automatizaciones", "Publicaciones", "Invitaciones" — taxonomía de desarrollador, no de dueño de negocio.
3. **Densidad sin lectura:** páginas de tarjetas apiladas donde ninguna responde una pregunta concreta; el usuario debe *escanear todo* para encontrar *algo*.
4. **Ausencia de estados intermedios consistentes:** hay skeletons en algunos módulos (UX.9 los añadió), pero al ser 82/107 páginas dinámicas sin `Suspense` por sección, la percepción dominante en conexiones LATAM (3G/4G irregular) es "pantalla en blanco… y de pronto todo".

## 4.2 Crítica por dimensión

| Dimensión | Veredicto | Detalle |
|---|---|---|
| Navegación | 🟠 | Cliente: aceptable (pocas secciones). Empresa: sobrecargada (~30 entradas planas). Regla violada: más de 7 items de primer nivel = el menú deja de ser un mapa y se vuelve un índice. |
| Jerarquía visual | 🟠 | Tipografía y tokens existen (DESIGN_SYSTEM.md), pero casi todas las páginas usan el mismo patrón `PageHeader + cards`, sin un "héroe" que declare qué importa. |
| Flujo / clics | 🟡 | Flujos clave razonables (registro por campaña: 2 pasos, bien). Pero: crear una promoción efectiva requiere entender tipos/horarios/stock/visibilidad sin plantillas a un clic → parálisis de formulario. |
| Claridad | 🟠 | Solapamiento conceptual Campañas/Estrategias/Automatizaciones (ver 4.1.2). |
| Fricción | 🟠 | El pico de fricción del producto: **pagar** (transferencia + foto del comprobante + esperar aprobación humana). Todo lo demás es secundario frente a esto. |
| Densidad/espacios | 🟡 | Correcto en general; tablas admin densas sin priorización de columnas. |
| Feedback visual | 🟡 | Toasts y estados de éxito existen; falta feedback de *progreso de negocio* (biografía del cliente: "llevas 3 visitas"). |
| Animaciones | 🟡 | Confetti en registro de campaña (bien). El resto, transiciones por defecto. |
| Pantallas vacías | 🟡 | Empty states existen en módulos renovados; desiguales en el resto. El empty state casi nunca *enseña* ("crea tu primera promo con esta plantilla"). |
| Errores | 🟡 | Alert genérico "No se pudo X". Sin código de referencia ni acción de recuperación. |
| Carga cognitiva | 🔴 | El panel de empresa es el punto crítico: demasiadas decisiones sin defaults con opinión. |

## 4.3 El nuevo sistema de experiencia: "Un camino, no un menú"

### Principio 1 — Cada rol tiene UNA pregunta; la home del rol la responde

| Rol | Su pregunta | La home debe mostrar |
|---|---|---|
| Cliente | "¿Qué tengo y qué puedo usar hoy?" | Wallet arriba (valor RD$ + beneficios activos), luego "cerca de ti / por vencer" |
| Empresa | "¿Está funcionando esto?" | Retorno del mes + 3 acciones recomendadas (no 30 links) |
| Empleado | "¿Escaneo y ya?" | Botón de escanear a pantalla completa + búsqueda manual |
| Superadmin | "¿Qué empresa necesita atención?" | Salud por tenant ordenada por riesgo |

### Principio 2 — Arquitectura de información del panel de empresa (rediseño)

```
ANTES (plano, ~30 items)                DESPUÉS (5 dominios + acción)
─────────────────────────               ──────────────────────────────
Dashboard                               ◉ Inicio        (retorno + recomendaciones)
Perfil / Publicaciones /                ◉ Clientes      (lista, segmentos, notas,
Promociones / Planes /                                  bandeja de acciones)
Campañas / Estrategias /                ◉ Ofertas       (promos + planes + plantillas)
Automatizaciones / Clientes /           ◉ Crecimiento   (invita y gana + publicaciones
Notificaciones / Referidos /                            + notificaciones + automatiz.)
Invitaciones / Pagos /                  ◉ Operación     (pagos + reportes + perfil
Reportes / Estadísticas /                               + config + equipo)
FAQ / Soporte / Config ...              ─ modo Simple (6 items) / Avanzado (todo)
```

### Principio 3 — Formularios con opinión

Todo formulario de creación (promoción, campaña, plan) abre con **plantillas** ("2x1 martes", "Primera visita gratis", "Meta 5 amigos") y el formulario completo queda como "personalizar". Las 36+ plantillas ya existen en el código (`lib/promotions/templates`) — es conectar, no construir.

### Principio 4 — Estados con narrativa

- **Empty:** siempre enseña + 1 CTA con plantilla.
- **Loading:** skeleton por sección con `Suspense` (no página en blanco).
- **Error:** qué pasó + qué hacer + referencia (`#MB-4821`) para soporte.
- **Éxito:** conecta con el siguiente paso ("Promo creada → compártela ahora").

### Principio 5 — Los tres momentos que hay que sobre-diseñar

La memoria del usuario se forma en picos. MembeGo tiene tres picos y debe gastarse el presupuesto de diseño ahí:

1. **Recibir un beneficio** (aparece en wallet): animación de "entrada a la wallet" + valor en RD$.
2. **Canjear** (escáner): pantalla de éxito a tamaño completo para cliente Y empleado, sonido, nombre del cliente.
3. **Lograr una meta** (referidos/retos): celebración (ya existe confetti en campañas — extenderlo como sistema, no como caso aislado).
